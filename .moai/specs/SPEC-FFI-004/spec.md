## 1. Environment

- **Language**: Rust 1.84+ (MSRV), Node.js 14+ (N-API v6+)
- **Binding Framework**: napi-rs 3.x (direct Rust-Node.js integration, C ABI bypassed)
- **Build Tool**: `@napi-rs/cli` (napi-rs standard build/publish tool)
- **N-API Version**: v6+ (stable ABI across Node.js versions, no recompilation needed)
- **Package Name**: `cypherlite` (npm), `cypherlite-node` (Rust crate)
- **CI**: GitHub Actions (existing 6 Rust jobs + new Node.js test jobs)
- **Target Platforms**: Linux (x86_64, aarch64), macOS (x86_64, aarch64), Windows (x86_64)
- **Feature Flags**: Rust feature flags (`subgraph`, `hypergraph`) propagated from `cypherlite-query`
- **Testing**: vitest 3.x
- **Type Checking**: TypeScript `.d.ts` definitions auto-generated by napi-rs
- **Workspace**: Cargo workspace at repository root (`crates/cypherlite-node/`)

## 2. Assumptions

- **A1**: SPEC-FFI-001 Assumption A2 establishes that Node.js bindings use napi-rs to directly wrap the Rust API, bypassing the C ABI (`cypherlite-ffi`).
- **A2**: napi-rs uses N-API (stable ABI) so a single native binary works across Node.js 14, 16, 18, 20, 22+ without recompilation.
- **A3**: `@napi-rs/cli` generates `package.json`, build scripts, and platform-specific npm packages following the napi-rs convention (`@cypherlite/linux-x64-gnu`, `@cypherlite/darwin-arm64`, etc.).
- **A4**: The `CypherLite` struct requires `&mut self` for `execute`, `execute_with_params`, and `begin`. napi-rs wrapping will use `Mutex<CypherLite>` internally to provide safe shared access from JavaScript.
- **A5**: CypherLite is an embedded database with no network I/O. A synchronous API is appropriate and idiomatic for this use case (similar to `better-sqlite3`). No async/Promise API is needed for v1.0.
- **A6**: Rust `Value` enum variants map to JavaScript native types (`null`, `boolean`, `number`, `string`, `Buffer`, `Array`). `NodeID`/`EdgeID` are exposed as `BigInt` values for 64-bit ID fidelity.
- **A7**: Feature-gated variants (Subgraph, Hyperedge, TemporalNode) are conditionally compiled in the napi-rs module using `#[cfg(feature = "...")]`.
- **A8**: The npm package will be published as `cypherlite`. Platform-specific native binaries are distributed via `optionalDependencies` packages following the napi-rs convention.

## 3. Requirements

### 3.1 Crate and Package Setup

**REQ-FFI-NODE-001** (Ubiquitous)
The `cypherlite-node` crate shall be located at `crates/cypherlite-node/` and be a member of the Cargo workspace. The crate shall produce a `cdylib` output for Node.js addon loading.

**REQ-FFI-NODE-002** (Ubiquitous)
The crate `Cargo.toml` shall declare `napi` and `napi-derive` as dependencies with `napi6` feature enabled, ensuring compatibility with Node.js 14+.

**REQ-FFI-NODE-003** (Ubiquitous)
The crate shall re-export all feature flags from `cypherlite-query` (e.g., `temporal-core`, `subgraph`, `hypergraph`) so that conditional compilation propagates correctly to napi-rs wrapper types.

**REQ-FFI-NODE-004** (Ubiquitous)
A `package.json` shall exist at the crate root (`crates/cypherlite-node/package.json`) declaring:
- `name`: `cypherlite`
- `napi.name`: `cypherlite`
- `napi.triples` with targets for Linux (x64, arm64), macOS (x64, arm64), and Windows (x64)
- `devDependencies`: `@napi-rs/cli`, `vitest`, `typescript`

**REQ-FFI-NODE-005** (Ubiquitous)
The build shall use `@napi-rs/cli` (`napi build --release`) to produce the native `.node` addon and auto-generate TypeScript `.d.ts` definitions.

**REQ-FFI-NODE-006** (Ubiquitous)
An `index.js` entry point shall exist at the crate root, implementing the napi-rs platform detection and binary loading pattern, selecting the correct native binary for the current OS/architecture.

**REQ-FFI-NODE-007** (Ubiquitous)
The package shall export the following public API:
- `open(path: string, config?: DatabaseConfig): Database`
- `version(): string`
- `features(): string`
- `CypherLiteError` error class
- TypeScript type definitions for all exported types

### 3.2 Database Lifecycle

**REQ-FFI-NODE-010** (Event-Driven)
**When** a JavaScript caller invokes `cypherlite.open(path)`, **then** the system shall create a `CypherLite` instance with default `DatabaseConfig` (using the provided path) and return a `Database` object, or throw `CypherLiteError` on failure.

**REQ-FFI-NODE-011** (Event-Driven)
**When** a JavaScript caller invokes `cypherlite.open(path, { pageSize: 4096, cacheCapacity: 1024 })`, **then** the system shall create a `CypherLite` instance with the specified configuration parameters.

**REQ-FFI-NODE-012** (Event-Driven)
**When** a JavaScript caller invokes `db.close()`, **then** the system shall drop the internal `CypherLite` instance, flushing all pending writes, and mark the `Database` handle as closed.

**REQ-FFI-NODE-013** (Unwanted)
The system shall not cause a crash or segfault when `close()` is called multiple times on the same `Database` handle; subsequent calls shall be no-ops.

**REQ-FFI-NODE-014** (Unwanted)
The system shall not allow operations on a closed `Database`; all methods shall throw `CypherLiteError` with message `"Database is closed"` if invoked after `close()`.

### 3.3 Query Execution

**REQ-FFI-NODE-020** (Event-Driven)
**When** a JavaScript caller invokes `db.execute(query)`, **then** the system shall acquire the internal Mutex, execute the Cypher query, and return a `Result` object, or throw `CypherLiteError`.

**REQ-FFI-NODE-021** (Event-Driven)
**When** a JavaScript caller invokes `db.execute(query, { name: "Alice" })`, **then** the system shall convert the JavaScript object to a Rust `HashMap<String, Value>` using the type mapping (REQ-FFI-NODE-060), execute the parameterized query, and return a `Result` object.

**REQ-FFI-NODE-022** (Unwanted)
The system shall not accept unsupported parameter value types; it shall throw `TypeError` with a descriptive message listing the unsupported type and the supported types.

### 3.4 Transaction Support

**REQ-FFI-NODE-030** (Event-Driven)
**When** a JavaScript caller invokes `db.begin()`, **then** the system shall create a `Transaction` object wrapping the Rust transaction state, or throw `CypherLiteError` if another transaction is already active.

**REQ-FFI-NODE-031** (Event-Driven)
**When** a JavaScript caller invokes `tx.execute(query)`, **then** the system shall execute the query within the transaction context and return a `Result` object.

**REQ-FFI-NODE-032** (Event-Driven)
**When** a JavaScript caller invokes `tx.execute(query, { name: "Alice" })`, **then** the system shall execute the parameterized query within the transaction context.

**REQ-FFI-NODE-033** (Event-Driven)
**When** a JavaScript caller invokes `tx.commit()`, **then** the system shall commit all changes, invalidate the `Transaction` handle, and return `undefined`.

**REQ-FFI-NODE-034** (Event-Driven)
**When** a JavaScript caller invokes `tx.rollback()`, **then** the system shall discard all changes, invalidate the `Transaction` handle, and return `undefined`.

**REQ-FFI-NODE-035** (Unwanted)
The system shall not allow operations on a consumed `Transaction`; `execute`, `commit`, and `rollback` shall throw `CypherLiteError` with message `"Transaction is closed"` after commit or rollback.

### 3.5 Result Access

**REQ-FFI-NODE-040** (Event-Driven)
**When** a JavaScript caller accesses `result.columns`, **then** the system shall return a `string[]` containing the column names of the result set.

**REQ-FFI-NODE-041** (Event-Driven)
**When** a JavaScript caller accesses `result.length`, **then** the system shall return the number of rows in the result set.

**REQ-FFI-NODE-042** (Event-Driven)
**When** a JavaScript caller invokes `result.get(index)`, **then** the system shall return a `Row` object for the given index, or throw `RangeError` if out of bounds.

**REQ-FFI-NODE-043** (Event-Driven)
**When** a JavaScript caller iterates `for (const row of result)`, **then** the system shall yield `Row` objects one by one (`Symbol.iterator` protocol).

**REQ-FFI-NODE-044** (Ubiquitous)
The `Result` object shall support `toArray()` returning a `Row[]` array of all rows.

**REQ-FFI-NODE-045** (Ubiquitous)
The `Result` object shall support `toString()` returning a human-readable summary: `[CypherLiteResult columns=[...] rows=N]`.

### 3.6 Row Value Access

**REQ-FFI-NODE-050** (Event-Driven)
**When** a JavaScript caller accesses `row.columnName` (property access by column name) or `row["column.name"]` (bracket notation), **then** the system shall return the value for the named column converted to a JavaScript type, or `undefined` if the column does not exist.

**REQ-FFI-NODE-051** (Event-Driven)
**When** a JavaScript caller accesses `row[index]` (numeric index), **then** the system shall return the value at the given column index, or `undefined` if out of bounds.

**REQ-FFI-NODE-052** (Ubiquitous)
The `Row` object shall be a plain JavaScript object with:
- String keys for column names (e.g., `{ "n.name": "Alice", "n.age": 30 }`)
- Numeric keys for positional access (e.g., `{ 0: "Alice", 1: 30 }`)
- `length` property equal to the number of columns

**REQ-FFI-NODE-053** (Event-Driven)
**When** a JavaScript caller invokes `row.toObject()`, **then** the system shall return a plain JavaScript object with column-name keys only (no numeric keys, no `length`).

**REQ-FFI-NODE-054** (Ubiquitous)
The `Row` object shall support `Object.keys(row)` returning column names (string keys only, excluding numeric indices).

### 3.7 Value Type Mapping

**REQ-FFI-NODE-060** (Ubiquitous)
The system shall map Rust `Value` enum variants to JavaScript types as follows:

| Rust Value Variant     | JavaScript Type       | Notes                                    |
| ---------------------- | --------------------- | ---------------------------------------- |
| `Value::Null`          | `null`                |                                          |
| `Value::Bool(b)`       | `boolean`             |                                          |
| `Value::Int64(i)`      | `number`              | Safe integer range (-2^53+1 to 2^53-1)  |
| `Value::Float64(f)`    | `number`              |                                          |
| `Value::String(s)`     | `string`              |                                          |
| `Value::Bytes(b)`      | `Buffer`              | Node.js Buffer                           |
| `Value::List(l)`       | `Array`               | Recursive conversion of elements         |
| `Value::Node(id)`      | `BigInt`              | NodeID as BigInt for 64-bit fidelity     |
| `Value::Edge(id)`      | `BigInt`              | EdgeID as BigInt for 64-bit fidelity     |
| `Value::DateTime(dt)`  | `Date`                | Converted from ms-since-epoch            |

**REQ-FFI-NODE-061** (State-Driven)
**While** the `subgraph` feature is enabled, the system shall additionally support:

| Rust Value Variant          | JavaScript Type   |
| --------------------------- | ----------------- |
| `Value::Subgraph(id)`       | `BigInt`          |

**REQ-FFI-NODE-062** (State-Driven)
**While** the `hypergraph` feature is enabled, the system shall additionally support:

| Rust Value Variant             | JavaScript Type                 |
| ------------------------------ | ------------------------------- |
| `Value::Hyperedge(id)`         | `BigInt`                        |
| `Value::TemporalNode(id, ts)` | `{ nodeId: BigInt, timestamp: BigInt }` |

**REQ-FFI-NODE-063** (Ubiquitous)
The JavaScript-to-Rust parameter conversion for `execute(query, params)` shall support the following JavaScript types:

| JavaScript Type    | Rust Value                     |
| ------------------ | ------------------------------ |
| `null`             | `Value::Null`                  |
| `undefined`        | `Value::Null`                  |
| `boolean`          | `Value::Bool`                  |
| `number`           | `Value::Int64` or `Value::Float64` (integer check via `Number.isInteger`) |
| `string`           | `Value::String`                |
| `Buffer`           | `Value::Bytes`                 |
| `Array`            | `Value::List` (recursive)      |
| `Date`             | `Value::DateTime` (to ms UTC)  |
| `BigInt`           | `Value::Int64` (range check)   |

**REQ-FFI-NODE-064** (Unwanted)
The system shall not silently convert unsupported JavaScript types (e.g., objects, functions, symbols); it shall throw `TypeError` with a descriptive message.

### 3.8 Error Handling

**REQ-FFI-NODE-070** (Ubiquitous)
The system shall define a custom `CypherLiteError` class extending JavaScript `Error`, exposed in the `cypherlite` module.

**REQ-FFI-NODE-071** (Ubiquitous)
The `CypherLiteError` class shall carry:
- `message` property: human-readable error description (inherited from `Error`)
- `code` property: string error code categorizing the error type (e.g., `"IoError"`, `"ParseError"`, `"TransactionConflict"`)

**REQ-FFI-NODE-072** (Event-Driven)
**When** a Rust `CypherLiteError` is returned from any API call, the system shall convert it to a JavaScript `CypherLiteError` with the appropriate message and code, and throw it.

**REQ-FFI-NODE-073** (Event-Driven)
**When** a JavaScript caller passes an invalid argument type to any method, the system shall throw standard JavaScript `TypeError` (not `CypherLiteError`).

### 3.9 Thread Safety

**REQ-FFI-NODE-080** (Ubiquitous)
The `Database` object shall use `Mutex<CypherLite>` internally to ensure safe access. napi-rs manages the JavaScript-to-Rust thread boundary; the Mutex ensures Rust-side exclusivity.

**REQ-FFI-NODE-081** (Ubiquitous)
The `Transaction` object shall NOT be safe to share between worker threads. It shall be documented as main-thread only.

**REQ-FFI-NODE-082** (Ubiquitous)
The `Result` and `Row` objects shall be plain JavaScript values after construction (no Rust references), safe to pass between contexts.

### 3.10 Library Info Functions

**REQ-FFI-NODE-090** (Event-Driven)
**When** a JavaScript caller invokes `cypherlite.version()`, **then** the system shall return the CypherLite version string (e.g., `"1.1.0"`).

**REQ-FFI-NODE-091** (Event-Driven)
**When** a JavaScript caller invokes `cypherlite.features()`, **then** the system shall return a comma-separated string of enabled Rust feature flags (e.g., `"temporal-core,subgraph"`).

### 3.11 TypeScript Definitions

**REQ-FFI-NODE-100** (Ubiquitous)
The package shall include a `index.d.ts` TypeScript definition file providing complete type annotations for all exported functions, classes, and interfaces.

**REQ-FFI-NODE-101** (Ubiquitous)
The TypeScript definitions shall be auto-generated by napi-rs `#[napi]` macro annotations during the build process.

**REQ-FFI-NODE-102** (Ubiquitous)
The TypeScript definitions shall include:
- `DatabaseConfig` interface with optional `pageSize` and `cacheCapacity` fields
- `Database` class with `execute`, `begin`, `close` methods
- `Transaction` class with `execute`, `commit`, `rollback` methods
- `Result` class with `columns`, `length`, `get`, `toArray`, `[Symbol.iterator]`
- `Row` type as an intersection of named properties and indexed access
- `CypherLiteError` class with `code` property
- `open`, `version`, `features` function signatures

The npm package structure shall be:

**REQ-FFI-NODE-111** (Ubiquitous)

## 4. Non-Functional Requirements

**REQ-FFI-NODE-NFR-001** (Performance)
napi-rs function call overhead shall not exceed 1 microsecond per call (excluding the actual database operation). The sync API avoids Promise/microtask overhead.

**REQ-FFI-NODE-NFR-002** (Memory)
The napi-rs layer shall not perform hidden heap allocations beyond those documented in the API. Rust-owned data shall be converted to JavaScript-owned objects at return time; no dangling references across the FFI boundary.

**REQ-FFI-NODE-NFR-003** (Safety)
Every `unsafe` block in the napi-rs crate shall have a `// SAFETY:` comment documenting the invariants. The goal is zero `unsafe` usage, as napi-rs handles the raw N-API boundary.

**REQ-FFI-NODE-NFR-004** (Compatibility)
The package shall build and pass tests on Linux (x86_64), macOS (x86_64, aarch64), and Windows (x86_64) with Node.js 14, 18, 20, and 22.

**REQ-FFI-NODE-NFR-005** (Idiomatic JavaScript)
The API shall follow JavaScript/Node.js conventions: sync methods for embedded DB operations (no unnecessary Promises), `Symbol.iterator` for result traversal, property access for row values, `Error` subclass for errors.

**REQ-FFI-NODE-NFR-006** (Install)
The package shall be installable via `npm install cypherlite` (from prebuilt binaries) and `npm run build` (for development from source).

## 5. Testing Requirements

**REQ-FFI-NODE-TEST-001** (Ubiquitous)
The crate shall include Rust unit tests verifying napi-rs wrapper correctness (value conversion, error mapping, Mutex behavior).

**REQ-FFI-NODE-TEST-002** (Ubiquitous)
The package shall include vitest integration tests (`__test__/`) exercising the complete API lifecycle: `open -> execute -> iterate results -> close`.

**REQ-FFI-NODE-TEST-003** (Ubiquitous)
vitest tests shall cover all error conditions: closed database, closed transaction, invalid parameter types, parse errors, execution errors, transaction conflicts.

**REQ-FFI-NODE-TEST-004** (Ubiquitous)
vitest tests shall verify that all Value type conversions round-trip correctly (JavaScript -> Rust -> JavaScript).

**REQ-FFI-NODE-TEST-005** (Ubiquitous)
vitest tests shall verify iterator protocol: `for...of result`, `result.length`, `result.get(i)`, `result.toArray()`, row property access, row index access.

**REQ-FFI-NODE-TEST-006** (Event-Driven)
**When** the `subgraph` or `hypergraph` features are enabled during build, vitest tests shall verify the corresponding feature-gated value types.

**REQ-FFI-NODE-TEST-007** (Ubiquitous)
TypeScript definitions shall be validated using `tsc --noEmit` against the test files to ensure annotation completeness and correctness.

## 7. Out of Scope

- Python bindings via PyO3 (SPEC-FFI-003)
- Go bindings via CGo (SPEC-FFI-002)
- Async/Promise API (synchronous API only in v1.0; embedded DB has no network I/O)
- Worker Threads sharing Database across threads
- Plugin system exposure (ScalarFunction, IndexPlugin, Serializer, Trigger)
- StorageEngine direct access (consumers use CypherLite facade only)
- Connection pooling or ORM-like abstractions
- Publishing to npm registry (build infrastructure only; publish is a separate workflow)
- C ABI interaction (this SPEC bypasses cypherlite-ffi entirely)
- Electron or browser support (Node.js only)
